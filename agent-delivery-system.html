01 / Workflow

CLAUDE.md 是交付协议，不是备忘录。

MuYu 项目把 Claude 定位成 CTO、PM 和最大用户，Codex 则承担高级工程师执行位。关键点是把每次开发变成可复用的流水线：先定义计划和验收，再实现，再验证，再发布。

计划和设计用户说“开始”后，计划进入 tasks/todo.md；含前端改动时必须产出 UI 规范。
计划审查/codex-plan-review 审查影响范围、接口消费方、存量兼容和闭环测试点。
代码执行/codex-exec 按 task 执行，Claude 不直接跳过流程写代码。
静态和功能验证Claude 做编译与规范检查；Codex 通过模拟器、curl、日志和端到端路径做功能测试。
发布闭环/deploy-remote 合入 main 并做线上测试；用户确认后再 /testflight-deploy。
02 / Team

muyu-claw-team 把 Agent 变成团队。

这个仓库的核心原则很克制：一个 bot 映射一个 agent，不做自动路由；私聊、群聊和定时任务会话隔离；60 秒内的新消息只合并进尚未开始的 pending task。它解决的是“多人、多 bot、多任务如何稳定进入执行队列”的问题。

Runtime flow
bridge event → normalize → resolve bot/agent/session → enqueue → immediate ack → worker → workspace projection → sandbox → Claude/Codex runtime → final delivery。

State boundary
Redis 和 PostgreSQL 承接队列、会话、群组记录和持久化状态；runtime 权限边界交给选定沙箱。

03 / Infra A

cui-no-auth 解决“看不见任务”的问题。

Claude Code 长任务不应该只能在终端里盯着。cui-no-auth 通过浏览器展示会话列表、详情更新和 sub-agent 执行过程，团队成员可以一起看同一条任务进度。

Directory Discovery每 15 秒扫描 ~/.claude/projects/**，发现新的 *.jsonl 并建立 sessionId 到文件路径的索引。
File Polling对已知 session 比较 mtime 和 size，推断首页列表和详情页是否需要更新。
SSE Broadcaster通过 session_list_update 和 session_content_update 把变化推到浏览器。
04 / Infra B

model-steer-claude 解决“模型断档”的问题。

长任务真正怕的不是慢，而是 429、500、额度耗尽之后整条链路断掉。NB-Claude 架在 Claude Code 和模型 API 之间，把 fallback、错误分类、熔断探测和任务路由放进同一个本地代理。

Never Fallback
主模型不可用时自动降级到下一个 provider；错误分类区分临时限流、服务异常、内容安全和真实配额耗尽。

Task Routing
cr --route 把路由规则注入会话，简单任务、编码任务、架构任务可以切到不同模型等级。

判断标准很简单：能不能被交付、被观察、被恢复。

如果 AI 编程只是一次聊天，它很难稳定生产。这里的方向是把它变成一个有队列、有角色、有日志、有模型续航、有发布验收的工程系统。
